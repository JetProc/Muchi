# MUMU 서비스·사용 설명서

> 기준일: 2026-07-18
> 대상: MUMU를 사용하는 사람, 사용자 테스트 진행자, 로컬 MVP를 실행·검수하는 팀원

## 1. 서비스 소개

MUMU는 좋아했던 음악을 `곡을 들었던 순간`과 함께 보관하는 모바일 개인 아카이브다. 사용자는 음악을 빠르게 포착하고, 챕터와 하위 챕터로 취향을 구성하며, 검색과 회고를 통해 기록을 다시 펼칠 수 있다.

핵심 기록 단위는 `특정 챕터에 담긴 곡의 기억`이다. 같은 곡이라도 챕터마다 다음 내용을 다르게 기록할 수 있다.

- 이 음악을 내가 어떻게 느끼는지 설명하는 성격 문장
- 월 또는 계절과 선택 연도로 표현하는 기억 시기
- `운동할 때`, `첫 자취방에서`처럼 경험을 표현하는 복수 태그
- 감상 날짜마다 추가하는 메모

모든 기록은 계정 없이 현재 브라우저에 저장된다. 다른 기기나 브라우저와 자동 동기화되지 않으므로 JSON 백업으로 보관한다.

## 2. 시작하기

처음 열면 다음 세 가지 중 하나를 선택한다.

| 선택 | 결과 |
|---|---|
| 샘플 기록 둘러보기 | 준비된 챕터와 기억을 보면서 기능을 탐색한다. |
| 빈 아카이브로 시작 | 샘플을 제거하고 아무 기록이 없는 상태에서 시작한다. |
| 내 첫 곡 기록하기 | 곧바로 RECORD 화면으로 이동한다. |

처음 사용하는 경우 `샘플 기록 둘러보기`로 챕터와 곡의 기억 구조를 확인한 뒤 설정에서 샘플만 제거하는 방법이 가장 안전하다.

## 3. 기본 내비게이션

### 하단 메뉴

| 메뉴 | 용도 |
|---|---|
| 홈 | 최근 기억, 정리할 곡, 루트 챕터와 최근 기록을 다시 본다. |
| 챕터 | 내가 만든 챕터와 자동 월별 챕터를 탐색·관리한다. |
| 기록 | 곡을 검색하거나 음악 링크로 가져온다. |
| 검색 | 곡·아티스트·챕터·태그·기억 문장을 한 번에 찾는다. |

### 헤더 메뉴

- `보관함`: 아직 챕터를 정하지 않은 곡을 정리한다.
- `설정`: 모션, 회고, 태그, 백업과 초기화를 관리한다.
- 하위 화면의 `뒤로`: 이전 챕터·단계 또는 상위 화면으로 돌아간다.

## 4. 첫 음악 기록 만들기

### 4.1 곡 찾기

하단의 `기록`을 누른다.

#### 곡명·아티스트로 검색

1. 검색창에 한 글자 이상의 곡명 또는 아티스트를 입력한다.
2. `검색`을 누른다.
3. 결과에서 원하는 곡의 `기록`을 누른다.
4. 결과가 많으면 아래로 스크롤할 때 10곡씩 이어서 나타난다.

검색은 iTunes의 한국 음악 카탈로그를 사용하며 인터넷 연결이 필요하다.

#### 공유 링크로 기록

1. RECORD 상단의 `공유 링크로 기록`을 누른다.
2. HTTPS 음악 링크를 붙여 넣는다.
3. `링크 가져오기`를 누른다.
4. 메타데이터가 부족하면 곡명·아티스트·앨범을 확인하거나 직접 입력한다.

| 서비스 | 지원 링크 | 메타데이터 동작 |
|---|---|---|
| Apple Music | 곡 또는 앨범 속 곡 링크 | iTunes 조회로 제목·아티스트·앨범·아트워크·미리듣기를 자동으로 가져올 수 있다. |
| Spotify | `open.spotify.com/track/...` | 제목·아트워크를 제안하지만 아티스트 정보가 부족해 수동 확인이 필요할 수 있다. |
| YouTube / YouTube Music | 일반 영상·뮤직 영상 링크 | 공식 oEmbed 범위에서 제목·채널·썸네일을 가져온다. |
| Melon | 곡 상세·가사 링크 | 링크를 식별한 뒤 곡명과 아티스트를 수동 입력한다. |

지원하지 않는 도메인, HTTP 링크, 플레이리스트·앨범 자체 링크는 곡으로 가져오지 못할 수 있다.

### 4.2 기록 방식 선택

곡을 고른 뒤 지금 남길 수 있는 정도를 선택한다.

- **곡만 기록**: 지금은 곡만 보관함에 남기고 키워드와 챕터 선택을 미룬다.
- **키워드로 기록**: `운동할 때`, `스무 살 여름`처럼 미래의 내가 찾을 말 하나 이상으로 곧바로 개인 기록을 완성한다.

키워드 기록은 별도의 사용자 챕터처럼 노출되지 않는다. 기록 후 `상세 더 남기기`로 성격·시기·날짜별 메모를 추가하거나, `챕터로 옮기기`로 플레이리스트형 수동 챕터에 정리할 수 있다.

RECORD에 입력한 검색어와 링크·수동 메타데이터 초안은 현재 탭의 세션 저장소에 임시 보존된다.

### 4.3 기억 확장

키워드로 기록한 뒤 상세 기록을 선택하거나 챕터 속 곡을 열면 MEMORY 화면으로 이동한다. 처음에는 태그 선택만 보이고, `더 남기기`를 펼치면 나머지 상세 필드가 나타난다.

| 입력 | 필수 여부 | 제한·설명 |
|---|---|---|
| 이 음악의 성격 | 선택 | 최대 100자. 장르명보다 개인적인 해석을 적을 수 있다. |
| 기억한 시기 | 선택 | 월 또는 계절을 고르고 연도는 비워둘 수 있다. |
| 태그 | 기록 시 1개 이상 | 한 기억에 최대 20개. 카테고리 없이 상황·시기·관계·기억을 짧은 문장으로 적는다. |
| 메모 | 선택 | 최대 1,000자. 감상 날짜별로 새로운 메모를 계속 추가할 수 있다. |

새 태그를 MEMORY에서 만들더라도 `키워드 기록`을 누르기 전에는 전체 아카이브에 반영되지 않는다. 성격·시기·메모는 모두 선택이며, 새 기록의 시기와 감상 날짜는 현재 연·월과 날짜로 자동 입력된다.

## 5. 같은 곡을 여러 순간으로 기록하기

MUMU에서 곡 정보와 곡의 기억은 서로 다른 데이터다. 여기서 같은 곡은 현재 `같은 음악 제공자와 제공자 곡 ID`로 가져온 항목을 뜻한다.

- 제목·아티스트·앨범 같은 곡 정보는 하나를 공유한다.
- 성격·시기·태그·메모는 `특정 챕터 안의 곡`에 기록한다.
- 같은 곡을 같은 챕터에 다시 넣으면 기존 기억을 연다.
- 같은 곡을 다른 챕터에 넣으면 비어 있는 새 기억을 만든다.

같은 음원을 Apple Music과 Spotify처럼 서로 다른 서비스 링크로 각각 가져오면 현재는 별도 곡으로 기록된다. 서비스 사이의 동일 음원을 자동으로 찾아 합치는 기능은 제공하지 않는다.

예를 들어 The Volunteers의 `Radio`를 `새벽 드라이브`와 `2018년 겨울`에 각각 기록해 서로 다른 태그와 메모를 남길 수 있다. 한쪽 기록을 바꿔도 다른 쪽은 바뀌지 않는다.

MEMORY의 `다른 챕터에도 기록`을 이용하면 현재 곡을 다른 수동 챕터에도 남길 수 있다.

## 6. 보관함 사용하기

보관함은 기록을 완성할 시간이 없을 때 쓰는 임시 대기 공간이다.

1. RECORD에서 곡을 찾은 뒤 `곡만 기록`을 선택한다.
2. 헤더의 `보관함` 또는 HOME의 `정리 대기`를 연다.
3. 곡의 `챕터 고르기`를 누른다.
4. 기존 수동 챕터를 선택하거나 새 챕터를 만든다.
5. 챕터로 이동한 곡의 기억을 작성한다.

`목록 관리`에서는 보관함에서 곡을 제거할 수 있다. 다른 챕터나 월별 챕터에서 여전히 참조하는 곡 기록은 유지된다.

## 7. 챕터 관리하기

### 7.1 수동 챕터

CHAPTERS의 `내가 만든 챕터` 탭에서 다음 작업을 할 수 있다.

- 새 챕터 생성
- 최근 활동, 이름, 곡 수 기준 정렬
- 앨범 커버 모음으로 챕터 구분
- 챕터 정보·분위기 색상 수정
- 챕터 삭제

챕터 이름은 최대 40자, 설명은 최대 200자다.

### 7.2 하위 챕터

챕터 상세의 `하위 챕터`에서 더 작은 장면을 만들 수 있다.

- 하위 챕터는 여러 단계로 중첩할 수 있다.
- 상단 경로에서 상위 챕터로 돌아갈 수 있다.
- 부모 챕터는 하위 챕터의 곡을 자동으로 합치지 않는다.
- 부모 챕터를 삭제하면 부모에 직접 담긴 곡 기억은 삭제되지만, 직접 하위 챕터는 한 단계 위로 이동해 남는다.

### 7.3 챕터 속 곡 관리

수동 챕터 상세에서 `곡 관리`를 켜면 곡 순서를 위·아래로 바꾸거나 현재 챕터에서 삭제할 수 있다. 삭제는 해당 챕터의 성격·태그·메모만 제거하며 다른 챕터의 같은 곡에는 영향을 주지 않는다.

## 8. 자동 월별 챕터

곡을 처음 포착하면 등록일을 기준으로 `YYYY년 M월` 챕터가 자동 생성된다.

- `월별 챕터` 탭에서 확인한다.
- 곡을 보관함에만 기록해도 해당 월에 자동 기록된다.
- 처음 등록한 날짜는 같은 곡을 다시 포착해도 유지된다.
- 월별 챕터는 수동 기록 대상으로 선택할 수 없다.
- 이름 수정, 삭제, 곡 기록, 하위 챕터 생성은 제공하지 않는다.
- 월별 상세는 등록한 음악을 다시 확인하는 읽기 전용 화면이다. 곡의 성격·태그·메모는 개인 챕터 또는 키워드 기록에서 남긴다.

월별 챕터는 `언제 이 곡을 MUMU에 등록했는지`를 보여주는 자동 목차이고, 사용자가 만든 챕터는 `이 곡이 나에게 어떤 장면인지`를 표현하는 개인 분류다.

## 9. 검색하기

SEARCH에서는 다음 정보를 한 번에 찾는다.

- 곡명, 아티스트, 앨범, 장르
- 챕터 이름과 설명
- 태그
- 이 음악의 성격
- 기억한 월·계절
- 메모 문장

여러 태그를 선택하면 `모두` 또는 `하나 이상` 조건을 고를 수 있다. 같은 곡이 여러 챕터에 있으면 각 챕터의 기억이 별도 결과로 나타난다. 월별 챕터는 등록일을 보는 읽기 전용 목차이므로 통합 검색 결과와 MEMORY 편집 화면에는 나타나지 않는다.

태그 조건이 없을 때는 아직 분류하지 않은 보관함의 곡도 검색 결과에 포함된다.

## 10. 음악 회고 사용하기

HOME의 `이맘때의 음악`에서 RECAP으로 이동한다.

| 방식 | 기준 |
|---|---|
| 이맘때의 음악 | 과거 같은 달·계절로 기록한 기억 또는 곡의 기억을 만든 과거 같은 월 |
| 지난 계절의 나 | 기억 시기와 기록일을 기준으로 정렬한 타임라인 |
| 무작위 기억 | 기록한 기억 중 무작위 선택 |

설정에서 `이맘때의 음악`을 끄면 회고 결과를 생성하지 않는다. 현재 리캡은 기록 목록을 다시 펼치는 기능이며 통계 대시보드나 공유용 소책자 이미지는 제공하지 않는다.

## 11. 태그 관리하기

헤더 또는 설정의 태그 진입점에서 전체 태그 라이브러리를 연다. 첫 화면은 관리보다 음악 재발견을 우선한다.

- 키워드별 기록된 곡 수 확인과 음악 그룹 열기
- 태그 이름 검색
- 카테고리 없이 모든 태그를 하나의 개인 언어로 관리
- 쉼표·세미콜론·줄바꿈으로 여러 태그 일괄 추가
- 태그 이름 수정
- 사용 중·미사용 수 확인
- 태그 삭제

태그는 음악의 객관적인 속성보다 `운동할 때`, `과거에 좋아했던 음악`, `첫 자취방에서`처럼 음악을 다시 찾고 싶은 순간을 기록하는 용도다. 외부 서비스의 장르, 감상 날짜와 등록일은 태그와 분리된 정보로 유지한다.

키워드 그룹에서는 같은 곡을 한 행으로 합쳐 보여준다. 같은 곡이 여러 챕터에 있다면 `N개 기억`을 펼쳐 각 챕터의 서로 다른 메모와 태그를 독립적으로 열 수 있다.

사용 중인 태그를 삭제하면 해당 태그가 연결된 모든 곡 기억에서도 함께 제거되므로 확인 문구를 읽고 진행한다.

## 12. 미리듣기와 원본 서비스

미리듣기 URL이 있는 곡은 MEMORY에서 최대 30초까지 들을 수 있다.

- 미니 플레이어를 누르면 전체 플레이어를 연다.
- 전체 플레이어의 `닫기`는 미니 플레이어로 축소한다.
- `플레이어 종료`는 재생 상태와 미니 플레이어를 모두 닫는다.
- `원본 열기`는 저장된 음악 서비스 링크로 이동한다.

미리듣기 링크가 만료되거나 브라우저가 재생을 막아도 챕터, 태그와 메모는 삭제되지 않는다.

## 13. 공개 챕터 탐색 프로토타입

하단의 `탐색`에서는 내 아카이브와 곡 구성이 닿는 가상 아카이버의 공개 챕터를 둘러볼 수 있다. 추천 근거는 태그보다 챕터 안에 함께 담긴 곡과 내 기록의 겹침을 우선한다.

- 공개 챕터에서는 곡 목록과 작성자가 공개한 메모·태그를 함께 읽을 수 있다.
- 곡은 보이지만 `기록이 비공개입니다`라고 표시되는 경우에는 메모와 태그를 볼 수 없다.
- 마음에 드는 챕터는 `플레이리스트로 듣기`를 눌러 기존 3단계 내보내기 흐름으로 이어 들을 수 있다.
- 프로필을 팔로우하고 챕터에 좋아요를 남길 수 있다. 팔로우한 프로필의 새 공개 챕터는 탐색 화면의 활동 버튼에서 확인한다.

현재 공개 프로필·챕터·활동은 시드 데이터이며, 팔로우·좋아요·읽음 상태만 현재 브라우저에 저장된다. 실제 사용자 계정이나 서버 알림은 연결하지 않는다.

## 14. 플레이리스트 내보내기 미리보기

곡이 담긴 챕터 상세에서 Apple Music, Spotify 또는 YouTube Music 플레이리스트 흐름을 열 수 있다.

1. 내보낼 곡과 플레이리스트 이름을 확인한다.
2. 대상 음악 서비스를 고른다.
3. 곡 매칭 결과를 검토하고 제외할 곡을 정한다.
4. `내보내기 준비 완료` 후 대상 서비스 홈페이지를 연다.

중요: 현재 매칭 결과는 실제 데이터가 아닌 UI 검증용 모의 결과다. 서비스 계정에 로그인하거나 실제 플레이리스트를 만들지 않으며, `다른 곡 찾기`도 현재 화면 안에서만 상태를 바꾼다.

## 15. 데이터 저장과 백업

### 저장 위치

- 아카이브 데이터는 현재 브라우저의 `localStorage`에 저장된다.
- 탐색의 팔로우·좋아요·활동 읽음 상태도 별도 `localStorage` 항목에 저장된다.
- RECORD의 작성 초안만 현재 탭의 `sessionStorage`에 임시 저장된다.
- 계정, 서버 데이터베이스, 기기 간 동기화는 없다.

따라서 브라우저 데이터 삭제, 비공개 모드 종료, 브라우저·기기 변경 시 기록이 사라질 수 있다. 민감한 개인정보는 메모에 입력하지 않는 것이 좋다.

### JSON 백업

1. `설정 → 내 기록 백업 → 내보내기`를 누른다.
2. `mumu-backup-YYYY-MM-DD.json` 파일을 안전한 위치에 보관한다.

### JSON 복원

1. `설정 → 내 기록 백업 → 불러오기`를 누른다.
2. MUMU JSON 백업 파일을 선택한다.
3. 현재 데이터를 교체한다는 확인을 읽고 승인한다.

복원과 초기화는 현재 아카이브 전체를 교체한다. 작업 전 기존 데이터를 먼저 내보내는 것을 권장한다.

### 샘플과 초기화

- `샘플만 제거`: 번들된 샘플과 샘플 챕터에 직접 담긴 곡 기억을 제거한다. 샘플 챕터에 사용자가 추가한 기록도 함께 사라지므로 먼저 다른 챕터로 옮긴다.
- `샘플로 초기화`: 현재 기록을 지우고 샘플 상태로 교체한다.
- `모든 기록 지우기`: 현재 기록을 지우고 빈 아카이브로 교체한다.

## 16. 오프라인 동작

배포 환경에서는 서비스 워커가 같은 출처의 앱 셸과 정적 자산을 제한적으로 저장한다.

| 오프라인에서 가능 | 오프라인에서 불가능 |
|---|---|
| 선캐시된 HOME·CHAPTERS 앱 셸 접근 | 새 iTunes 음악 검색 |
| 해당 화면에서 브라우저에 저장된 기록 열람 | 음악 링크 메타데이터 가져오기 |
| 이전 온라인 방문으로 캐시된 같은 출처 화면 접근 | 30초 미리듣기와 외부 서비스 열기 |

SEARCH와 상세 화면은 기본 선캐시 대상이 아니므로 온라인 상태에서 한 번도 열지 않은 경로는 오프라인에서 열리지 않을 수 있다. 외부 앨범 아트와 오디오 파일은 서비스 워커 캐시에 저장하지 않는다. 로컬 개발 환경에서는 오래된 캐시 간섭을 막기 위해 서비스 워커를 해제한다.

## 17. 접근성과 모션

- 주요 터치 영역은 최소 44×44px다.
- 시스템 모션 설정 따르기, 모션 줄이기, 감성 모션 사용 중 하나를 선택한다.
- 모션 감소에서는 공유 이미지 확대와 과도한 이동을 줄인다.
- 다이얼로그와 전체 플레이어는 포커스를 내부에 유지하고 닫은 뒤 안전한 위치로 복귀한다.
- HOME 캐러셀은 스와이프, 키보드 좌우 화살표, 이전·다음 버튼을 지원하고 현재 항목을 음성으로 알린다.
- 오프라인, 저장 실패, 검색 실패는 상태·경고 문구로 전달한다.

## 18. 문제 해결

### 검색 또는 링크 가져오기가 되지 않음

- 인터넷 연결을 확인한다.
- 링크가 HTTPS인지 확인한다.
- Apple Music 곡, Spotify 트랙, YouTube 영상, Melon 곡 상세 링크인지 확인한다.
- 외부 서비스가 응답하지 않으면 잠시 후 다시 시도하거나 수동 입력을 사용한다.

### 미리듣기가 재생되지 않음

- 모든 곡이 미리듣기를 제공하지는 않는다.
- 브라우저의 자동 재생 정책이나 만료된 외부 링크로 실패할 수 있다.
- 원본 서비스 링크를 사용해 재생할 수 있으며 기록 데이터는 유지된다.

### 저장되지 않음 또는 보호 모드가 표시됨

- 브라우저의 사이트 저장소 사용이 차단되었는지 확인한다.
- 다른 탭에서 데이터가 바뀌었다면 최신 데이터를 확인한 뒤 다시 저장한다.
- 가능한 경우 원본 브라우저에서 먼저 JSON 백업을 만든다.
- 보호 모드에서 계속할 수 없다면 설정의 초기화가 필요할 수 있으나, 초기화 전 데이터 손실 가능성을 확인한다.

### 기록이 다른 기기에 보이지 않음

현재 기기 간 동기화 기능이 없다. 기존 브라우저에서 JSON을 내보내고 새 브라우저에서 불러온다.

## 19. 로컬 실행·검수

### 요구 사항

- Node.js `22.13.0` 이상
- npm

### 실행

```bash
npm install
npm run dev
```

명령이 출력하는 로컬 주소를 연다. 앱은 실제 모바일에서는 전체 화면을 사용하고, 넓은 브라우저에서는 최대 430px 모바일 프레임으로 표시된다. 별도의 데스크톱 UI는 제공하지 않는다.

### 검증

```bash
npm run lint
npm run typecheck
npm test
```

`npm test`에는 프로덕션 빌드가 포함된다. 주요 화면 렌더링, 모바일 레이아웃, 링크 허용 범위, 챕터 계층, 같은 곡의 맥락 독립성, 월별 분류, 검색과 태그 동작을 확인한다.

## 20. 기술·운영 범위

- UI: Next.js·React·TypeScript 기반 모바일 웹
- 개발·빌드: vinext, Vite, Wrangler
- 영속 데이터: 브라우저 `localStorage`; 현재 서버 DB 사용 없음
- 서버 역할: iTunes 검색과 지원 음악 링크 메타데이터를 같은 출처 API로 중계·정규화
- PWA: 앱 셸과 같은 출처 정적 자산의 제한적 오프라인 캐시
- 외부 리소스: 앨범 아트·미리듣기·음악 서비스 링크는 네트워크 의존

서비스 범위에는 계정·클라우드 동기화, 실제 사용자 공개 아카이브와 서버 기반 소셜 기능, 실제 플레이리스트 생성, 자체 스트리밍, 직접 음원 업로드, 네이티브 앱이 포함되지 않는다.

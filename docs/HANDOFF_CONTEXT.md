## 1. 현재 제품 정의

MUCHI는 좋아했던 음악을 잊기 전에 포착하고, 곡마다 다른 성격·시기·태그·메모를 붙여 나만의 챕터로 엮는 모바일 개인 음악 아카이브다.

핵심 문장은 다음과 같다.

> 플레이리스트를 만드는 앱이 아니라, 음악을 들었던 순간을 나만의 언어로 편집하는 아카이브.

현재 제품은 모바일 전용 로컬 우선 MVP다. 계정, 서버 저장소, 클라우드 동기화 없이 사용자의 브라우저 `localStorage`에 기록을 저장한다. 제품의 중심은 다른 사람의 플레이리스트 탐색이 아니라, 사용자가 자신의 음악 세계를 기록하고 다시 찾는 경험이다.

## 2. 중요한 방향 전환

초기 아이디어에는 큐브, 캐릭터, 3D 음악 세계 탐험이 있었다. 현재 구현과 문서에서는 이 방향을 버리고, 앨범 아트와 기억을 편집하는 모바일 개인 음악 매거진으로 정리했다.

현재 사용자에게 보이는 핵심 용어는 `챕터`다. 기존 내부 데이터 구조에는 호환성을 위해 `Cube`, `CubeTrack`, `muchi:*` 같은 이름이 일부 남아 있지만, 사용자-facing 문서와 UI에서는 챕터와 곡의 기억이라는 표현을 사용한다.

디자인 방향도 웜 아날로그 매거진에서 출발했지만, 현재 구현은 더 미니멀한 Apple식 모바일 앱 프레임으로 정리되어 있다. 핵심 원칙은 최대 430px 모바일 캔버스, 앨범 아트 중심, 절제된 타이포그래피, 텍스트 기반 하단 내비게이션이다.

## 3. 사용자 문제와 제품 가설

해결하려는 문제는 단순하다. 사용자는 좋아했던 음악을 스트리밍 앱의 좋아요나 플레이리스트에만 맡겨두다가, 시간이 지나면 그 음악을 왜 좋아했는지와 어떤 시절에 들었는지를 잊는다.

현재 제품 가설은 다음과 같다.

> 음악과 기억의 관계를 중요하게 여기는 사용자는 곡을 빠르게 포착한 뒤 챕터별로 성격·시기·복수 태그·메모를 독립 기록할 수 있다면, 평면적인 좋아요나 플레이리스트보다 더 오래 음악을 보존하고 다시 찾을 것이다.

초기 인터뷰에서 나온 주요 인사이트는 다음과 같다.

- 타깃층은 넓지 않지만 선명하다. 음악 기록 욕구가 있는 사람에게는 강한 수요가 있을 수 있다.
- 일반 사용자는 입력 귀찮음이 크다. 따라서 초반에는 모두를 위한 자동화보다 기록 자체를 가치로 느끼는 사용자를 우선해야 한다.
- 많은 사용자가 스트리밍 앱의 좋아요를 사실상 기록 도구로 쓰고 있다.
- 이 제품이 잘 작동하면 음악을 듣는 행위가 더 의식적인 기록 행위로 바뀔 가능성이 있다.
- 모바일 친화성이 중요하다. 실제 음악 감상과 포착은 대부분 모바일에서 일어난다.
- 태그 입력은 일반 사용자에게는 부담이지만, 핵심 타깃에게는 제품을 쓰는 이유가 될 수 있다.

## 4. 현재 구현된 핵심 기능

현재 MVP에서 구현된 주요 기능은 다음과 같다.

- iTunes KR 곡 검색
- Apple Music·YouTube Music 공유 링크 메타데이터 가져오기. Spotify 가져오기는 준비 중
- 메타데이터가 부족할 때 수동 보완
- 분류를 미룰 수 있는 보관함
- 수동 챕터, 하위 챕터, 자동 월별 챕터
- 챕터 안의 곡별 성격 문장, 기억 시기, 복수 태그, 메모
- 같은 곡을 여러 챕터에 넣었을 때 태그와 기억의 독립 유지
- 곡, 아티스트, 챕터, 태그, 기억 문장 통합 검색
- 태그 라이브러리 관리
- 이맘때, 타임라인, 무작위 방식의 회고
- 30초 미리듣기와 원본 음악 서비스 연결
- JSON 백업과 복원
- 샘플 기록 제거, 샘플 초기화, 전체 초기화
- 플레이리스트 내보내기 흐름 미리보기
- 모바일 전용 430px 앱 프레임
- 모션 감소, 포커스 관리, 제한적 오프라인 앱 셸

플레이리스트 내보내기는 현재 실제 외부 서비스에 플레이리스트를 만들지 않는다. 곡 선택, 서비스 선택, 모의 매칭, 서비스 열기까지의 사용자 흐름 검증용 UI다.

## 5. 핵심 데이터 규칙

MUCHI의 가장 중요한 규칙은 `곡 정보`와 `곡의 기억`을 분리하는 것이다.

- 같은 제공자와 같은 제공자 곡 ID는 하나의 곡 정보로 본다.
- 같은 곡을 같은 챕터에 다시 넣으면 기존 기억을 연다.
- 같은 곡을 다른 챕터에 넣으면 새로운 곡의 기억을 만든다.
- 성격, 시기, 태그, 메모는 챕터와 곡의 관계에 저장된다.
- 따라서 같은 곡이라도 챕터가 다르면 태그와 기억이 서로 영향을 주지 않는다.
- 같은 음원을 Apple Music과 Spotify처럼 다른 서비스에서 각각 가져오면 현재는 별도 곡으로 취급한다.

자동 월별 챕터는 사용자가 만든 챕터와 역할이 다르다. 월별 챕터는 음악을 처음 포착한 등록 월 기준의 자동 목차이고, 수동 챕터는 사용자가 직접 부여한 음악의 장면과 의미다.

## 6. 현재 UI/UX 기준

제품은 모바일만 기준으로 판단한다. 데스크톱 전용 레이아웃은 점검 대상이 아니다.

- 앱 최대 폭은 430px이다.
- 실제 모바일에서는 `100vw × 100dvh`를 사용한다.
- 넓은 브라우저에서는 430px 앱 캔버스를 가운데 표시한다.
- 앱 프레임은 `44px 헤더 / 내부 스크롤 콘텐츠 / 70px 하단 내비게이션` 구조다.
- 브라우저 문서 전체가 아니라 앱 내부 콘텐츠만 스크롤된다.
- 하단 내비게이션은 `HOME / CHAPTERS / ADD / SEARCH` 성격의 모바일 탭 구조를 유지한다.
- 챕터 목록은 항상 2열이다.
- 기억 화면은 항상 1열이다.
- 다이얼로그, 태그 시트, 미니 플레이어, 전체 플레이어, 토스트는 430px 프레임 밖으로 나가지 않아야 한다.

최근 수정된 UX 포인트는 다음과 같다.

- 루트 탭에서는 불필요한 상단 뒤로가기를 제거했다.
- 챕터 상세에서 하위 챕터 생성 UI를 추가했다.
- 챕터 상세의 우선순위는 `하위 챕터 → 수록곡 → 플레이리스트 내보내기`다.
- 전체 플레이어의 `닫기`는 미니 플레이어로 축소하고, `플레이어 종료`는 재생 상태를 완전히 닫는다.
- 플레이리스트 내보내기 로고는 원형 버튼 중앙에 맞도록 정렬했다.

## 8. 최근 큰 작업 내역

최근 작업의 큰 흐름은 다음과 같다.

1. 큐브·캐릭터·3D 월드 방향을 제거하고 모바일 개인 음악 매거진 방향으로 전환했다.
2. 제품 기획서와 서비스 설명서를 현재 구현 기준으로 다시 작성했다.
3. 모바일 전용 앱 프레임을 적용했다.
4. 데스크톱 레이아웃 점검을 중단하고, 넓은 화면에서도 430px 모바일 프레임만 보이게 했다.
5. 챕터 안에서 하위 챕터를 만들고 이동할 수 있는 UI/UX를 추가했다.
6. 전체 UI 깨짐을 점검하고 모바일 기준으로 스타일을 보정했다.
7. UX 리뷰를 거쳐 루트 탭 뒤로가기, 플레이어 종료, 챕터 상세 우선순위 등을 정리했다.
8. 플레이리스트 내보내기 로고 중앙 정렬 문제를 수정했다.
9. 쓰지 않는 스타터 파일, D1/Drizzle 예제, 기본 SVG, 비활성 레거시 테마를 정리했다.
10. 문서에서 이전 3D·큐브·캐릭터 기획 흔적을 줄이고 현재 서비스 내용에 집중하도록 수정했다.

## 9. 삭제·정리된 범위

최근 정리에서 다음 성격의 파일과 의존성을 제거했다.

- ChatGPT auth 스타터 파일
- Cloudflare D1/Drizzle 예제와 설정
- 사용하지 않는 `db/` 스키마
- 기본 Next/Vite SVG 에셋
- 사용하지 않는 LP 이미지
- Tailwind/PostCSS 설정과 관련 의존성
- Sites 빌드 플러그인의 불필요한 Drizzle 복사 로직
- Worker Env의 사용하지 않는 DB 바인딩

이 정리 후 `npm test`, `npm run lint`, `npm run typecheck`가 통과한 상태로 보고되었다.

## 10. 실행과 검증

로컬 실행:

```bash
npm install
npm run dev
```

검증:

```bash
npm run lint
npm run typecheck
npm test
```

`npm test`는 프로덕션 빌드와 `tests/rendered-html.test.mjs` 회귀 테스트를 함께 실행한다. 현재 테스트에는 모바일 프레임, 주요 화면 셸, 챕터·기억 데이터 규칙, 플레이리스트 UI, 백업·복원 관련 검사가 포함되어 있다.

## 11. 현재 확인된 제약과 위험

- 데이터는 브라우저 `localStorage`에만 저장된다. 브라우저 데이터 삭제나 기기 변경에 취약하다.
- 실제 계정, 클라우드 동기화, 서버 DB는 없다.
- 플레이리스트 내보내기는 실제 외부 서비스 API 연동이 아니다.
- Spotify와 Melon 링크 자동 가져오기는 아직 제공하지 않는다. 베타에서는 직접 검색으로 같은 곡을 찾는다.
- 서로 다른 음악 서비스의 같은 음원을 자동으로 병합하지 않는다.
- 오프라인 앱 셸은 제한적이다. 검색, 링크 가져오기, 미리듣기는 온라인 의존이다.
- `npm audit` 기준 취약점이 남아 있다는 보고가 있었다. 자동 수정은 의존성 업그레이드 범위라 보류했다.
- 출시 전 실제 iOS/Android 기기에서 주소창 축소, 키보드 노출, 안전 영역을 직접 확인해야 한다.

## 12. 다음 작업 추천

바로 이어서 작업한다면 우선순위는 다음이 합리적이다.

1. 실제 모바일 기기에서 첫 곡 추가, 챕터 생성, 기억 저장, 검색, 백업까지 전체 흐름을 검수한다.
2. 사용자 테스트에서 태그·성격·시기·메모 중 반복 사용되는 필드를 확인한다.
3. 보관함에서 챕터로 정리하는 흐름이 실제로 입력 부담을 줄이는지 확인한다.
4. 플레이리스트 내보내기를 실제 연동할지, 아니면 MVP에서는 계속 미리보기로 둘지 결정한다.
5. 로컬 저장 한계를 줄이기 위해 백업 안내를 강화할지, 계정·동기화를 다음 단계로 잡을지 결정한다.
6. 의존성 보안 경고를 별도 브랜치에서 업그레이드 범위로 점검한다.

## 13. 다른 계정에서 이어받을 때 첫 확인 순서

새 계정이나 새 Codex 작업에서 이어받을 때는 다음 순서로 확인한다.

1. `README.md`로 프로젝트 목적과 실행 방법을 확인한다.
2. `docs/MUCHI_PRODUCT_PLAN.md`로 현재 제품 방향을 확인한다.
3. `docs/MUCHI_SERVICE_GUIDE.md`로 실제 사용 흐름을 확인한다.
4. `docs/HANDOFF_CONTEXT.md`로 최근 결정과 작업 히스토리를 확인한다.
5. `npm install` 후 `npm run dev`로 앱을 실행한다.
6. 브라우저에서 430px 모바일 프레임 기준으로 HOME, ADD, CHAPTERS, MEMORY, SEARCH를 본다.
7. `npm run lint`, `npm run typecheck`, `npm test`로 현재 상태를 검증한다.

## 14. 작업 시 주의할 점

- 사용자-facing 용어는 `챕터`, `곡의 기억`, `보관함`, `회고`를 사용한다.
- 새 문서나 UI에서 큐브, 월드, 캐릭터, 3D 탐험을 다시 중심 개념으로 되돌리지 않는다.

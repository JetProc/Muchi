# MUMU 제품 기획서

> 기준일: 2026-07-18
> 제품 단계: 모바일 전용 · 로컬 우선 수요 검증 MVP
> 문서 기준: 현재 저장소에 실제 구현된 기능과 검증이 필요한 후속 가설을 분리해 기술한다.

## 1. 제품 요약

### 한 줄 정의

MUMU는 좋아했던 음악을 잊기 전에 포착하고, 곡마다 다른 성격·시기·경험 태그·메모를 붙여 나만의 챕터로 엮는 개인 음악 아카이브다.

### 핵심 문장

> 플레이리스트를 만드는 앱이 아니라, 음악을 들었던 순간을 나만의 언어로 편집하는 아카이브.

### 현재 제품의 형태

- 앨범 아트와 기억을 감각적으로 편집하는 **모바일 개인 음악 매거진**이다.
- 핵심 분류 단위는 `챕터`이며 챕터 안에 하위 챕터를 만들 수 있다.
- 내 음악을 기록하고 정리하며 다시 발견하는 경험에 집중한다.
- 데이터는 서버가 아닌 현재 브라우저에 저장되는 로컬 우선 MVP다.

## 2. 해결하려는 문제

음악 서비스의 좋아요와 일반 플레이리스트는 곡을 모으는 데에는 편리하지만, 사용자가 그 음악을 왜 좋아했는지와 어떤 시절에 들었는지를 충분히 남기지 못한다.

그 결과 다음 문제가 발생한다.

1. 좋아했던 곡을 따로 기록하지 않아 시간이 지나면 곡과 그 시절의 기억이 함께 잊힌다.
2. 플레이리스트는 보통 하나의 제목 아래 곡을 나열하므로 곡마다 다른 시기와 경험의 연결을 표현하기 어렵다.
3. 같은 곡이 `새벽 드라이브`와 `2018년 겨울`에 모두 속하더라도 각 순간의 의미를 독립적으로 기록하기 어렵다.
4. 알고리즘 추천은 새로운 곡을 발견하게 하지만, 사용자가 직접 만든 맥락을 통해 과거의 음악을 다시 찾게 하지는 못한다.
5. 세밀한 기록은 가치가 있지만 입력 부담이 크기 때문에 포착과 정리 사이의 마찰을 낮춰야 한다.

### 핵심 제품 가설

음악과 기억의 관계를 중요하게 여기는 사용자는 곡을 빠르게 포착한 뒤 챕터별로 성격·시기·경험 태그·메모를 독립 기록할 수 있다면, 평면적인 좋아요나 플레이리스트보다 더 오래 음악을 보존하고 다시 찾을 것이다.

## 3. 목표 사용자

### 1차 사용자

- 좋아했던 노래가 좋아요 목록 아래로 밀려 잊히는 경험을 자주 하는 사람
- 음악과 특정 시기·사람·장면의 연결을 중요하게 여기는 음악 애호가
- 플레이리스트 제목 하나보다 곡마다 다른 성격과 감정을 직접 분류하고 싶은 사람
- 알고리즘 추천보다 자신의 언어와 맥락으로 음악을 재발견하고 싶은 사람
- 기록을 위해 어느 정도의 수고를 감수할 의향이 있는 모바일 사용자

### 우선 대상이 아닌 사용자

- 음악을 기록하거나 다시 찾을 필요를 거의 느끼지 않는 사용자
- 자동 추천과 단순 좋아요만으로 충분한 사용자
- 실시간 소셜, 스트리밍, 협업 플레이리스트가 주목적인 사용자

초기에는 모든 사람의 입력 부담을 없애기보다, 기록 자체를 가치로 느끼는 선명한 타깃의 핵심 경험을 검증한다.

## 4. 사용자에게 제공할 가치

| 가치 | 사용자 결과 |
|---|---|
| 잊힘 방지 | 좋아했던 곡을 발견한 시점에 포착하고 나중에 정리할 수 있다. |
| 입체적 분류 | 한 곡에 성격, 시기, 여러 경험 태그와 메모를 함께 부여한다. |
| 맥락 독립성 | 같은 곡을 여러 챕터에 담아도 각 챕터의 기록이 섞이지 않는다. |
| 취향의 구조화 | 챕터와 하위 챕터로 음악 취향을 개인적인 목차처럼 구성한다. |
| 재발견 | 통합 검색과 회고를 통해 오래된 음악과 기억을 다시 만난다. |
| 소유감 | 외부 음악 링크가 실패해도 사용자가 쓴 기억과 분류는 남는다. |

## 5. 제품 원칙

1. **아카이브 우선**: 외부 서비스 연결보다 내 기록 생성과 재열람을 먼저 보여준다.
2. **곡보다 맥락**: 곡 원본 정보는 공유하되, 챕터 속 곡의 기억은 별도 레코드로 취급한다.
3. **빠르게 포착하고 나중에 확장**: 곡만 담거나 태그 하나만 남겨도 시작할 수 있고, 챕터·성격·시기·메모는 나중에 확장한다.
4. **자동 분류와 수동 세계를 구분**: 월별 챕터는 등록일 기반 기록이고 사용자의 수동 챕터를 대체하지 않는다.
5. **편집된 음악 세계**: 앨범 아트, 타이포그래피, 챕터 계층으로 개인의 음악 세계를 표현한다.
6. **모바일에서 완결**: 모든 핵심 기록 흐름을 최대 430px 화면 안에서 완료한다.
7. **외부 실패와 기록을 분리**: 검색·메타데이터·미리듣기 실패가 아카이브 데이터 손실로 이어지지 않게 한다.

| 챕터 | 사용자가 만든 음악 분류와 장면 |

## 7. 정보 구조와 핵심 화면

### 주요 내비게이션

하단 내비게이션은 `홈 / 챕터 / 기록 / 탐색 / 검색` 다섯 항목으로 구성한다. 보관함과 설정은 헤더에서, 회고는 홈의 연결을 통해 진입한다.

| 화면 | 역할 | 현재 핵심 기능 |
|---|---|---|
| HOME | 다시 펼쳐보는 개인 음악 표지 | 최근 기억, 정리 대기, 최근 태그, 태그 없는 곡, 수동 챕터 |
| RECORD | 음악 기록 | iTunes 검색, 공유 링크 가져오기, 곡만 기록 또는 태그 하나로 기록 |
| INBOX | 나중에 정리 | 미분류 곡을 수동 챕터로 이동하거나 제거 |
| CHAPTERS | 음악 세계의 목차 | 수동·월별 탭, 정렬, 2열 앨범 커버 목록, 새 챕터 |
| CHAPTER DETAIL | 한 장면의 구성 | 경로, 직접 하위 챕터, 곡 목록, 순서·삭제 관리, 플레이리스트 진입 |
| MEMORY | 곡의 맥락 편집 | 태그 우선 편집, 선택 상세 영역의 성격·월·계절·날짜별 메모, 챕터 이동 |
| DISCOVER | 공개 기록 디깅 | 곡 구성 유사도가 높은 공개 챕터, 공개 기록 열람, 팔로우·좋아요·새 챕터 활동 |
| PUBLIC CHAPTER | 타인의 챕터 열람 | 공개 메모·태그 열람, 비공개 기록 안내, 플레이리스트 내보내기 |
| SEARCH | 전체 아카이브 회수 | 텍스트 통합 검색, 다중 태그 필터, 태그별 곡 그룹과 독립 기억 펼치기 |
| RECAP | 과거 음악 재발견 | 이맘때, 타임라인, 무작위 기억 |
| TAGS | 태그로 음악 재발견 | 태그별 곡 수 탐색, 그룹 진입, 태그 생성·검색·수정·삭제 |
| SETTINGS | 표시·데이터 관리 | 모션, 회고, JSON 백업·복원, 샘플 제거, 초기화 |
| PLAYLIST | 외부 내보내기 흐름 검증 | 곡 선택, 서비스 선택, 모의 매칭, 서비스 열기 |

## 8. 핵심 사용자 흐름

### 흐름 A — 첫 곡을 기억으로 완성하기

1. 온보딩에서 샘플 탐색, 빈 아카이브, 첫 곡 기록 중 하나를 선택한다.
2. RECORD에서 곡명·아티스트를 검색하거나 지원 음악 링크를 붙여 넣는다.
3. `태그로 기록`을 선택하고 미래의 내가 찾을 말 하나를 남긴다.
4. 필요하면 완료 화면에서 MEMORY로 이어가 성격, 시기와 날짜별 메모를 추가한다.
5. 필요하면 수동 챕터로 옮기고, 태그 그룹이나 챕터에서 동일한 기록을 다시 찾는다.

### 흐름 B — 지금은 곡만 잃지 않기

1. RECORD에서 곡을 찾는다.
2. `곡만 기록`을 선택해 보관함에 남긴다.
3. HOME의 정리 대기 또는 INBOX에서 곡을 다시 연다.
4. 수동 챕터를 선택하고 곧바로 MEMORY를 작성한다.

### 흐름 C — 음악 세계를 계층화하기

1. CHAPTERS에서 루트 챕터를 만든다.
2. 챕터 상세에서 하위 챕터를 만든다.
3. 경로 표시를 통해 상위·하위 장면을 이동한다.
4. 곡은 정확히 담긴 챕터에만 속하며, 부모가 하위 챕터의 곡을 자동 합산하지 않는다.

### 흐름 D — 오래된 음악을 다시 찾기

1. HOME 또는 TAGS에서 `운동할 때` 같은 개인 태그를 고른다.
2. 같은 태그를 가진 음악은 곡 기준으로 한 번씩 보고, 각 곡 안에서 챕터별 독립 기억을 펼친다.
3. 더 구체적인 조건은 SEARCH에서 검색어와 여러 태그를 조합하고, RECAP에서는 시기 기반으로 다시 펼친다.

## 9. 핵심 데이터·비즈니스 규칙

### 곡과 기억

- 음악 제공자와 제공자 곡 ID가 같으면 하나의 곡 정보를 공유한다.
- 같은 제공자의 같은 곡 항목을 같은 챕터에 다시 담으면 기존 기록을 연다.
- 같은 제공자의 같은 곡 항목을 다른 챕터에 담으면 새로운 곡의 기억을 만든다.
- 같은 음원을 서로 다른 음악 서비스 링크로 가져오면 현재는 별도 곡으로 취급하며 자동 중복 통합하지 않는다.
- 성격·시기·태그·메모는 챕터-곡 관계에 기록되므로 서로 영향을 주지 않는다.
- 현재 입력 제한은 성격 100자, 메모 1,000자, 한 기억당 태그 20개다.

### 챕터 계층

- 챕터는 하위 챕터를 계속 만들 수 있으며 순환 구조는 허용하지 않는다.
- 부모 챕터는 하위 챕터의 곡을 자동으로 포함하지 않는다.
- 부모 챕터를 삭제하면 부모에 직접 담긴 기억은 삭제되고, 직접 하위 챕터는 한 단계 위로 이동해 보존된다.
- 챕터 안에서 곡을 삭제하면 해당 챕터의 기억만 제거하고 다른 챕터의 같은 곡은 유지한다.

### 월별 챕터

- 음악을 처음 포착하면 Asia/Seoul 등록일을 기준으로 `YYYY년 M월` 챕터가 자동 생성된다.
- 보관함에만 기록한 곡도 월별 챕터에는 등록된다.
- 월별 챕터는 수동 곡 배정, 이름 수정, 삭제, 곡 기록, 하위 챕터 생성을 허용하지 않는다.
- 비어 있는 월별 챕터는 목록에서 숨긴다.

### 태그

- 태그 정의는 전체 아카이브에서 재사용한다.
- 실제 선택 상태는 각 곡의 기억에 저장한다.
- 태그만 빠르게 남긴 기록은 숨겨진 `capture` 컨텍스트에 기록하고, 수동 챕터로 옮길 때 같은 기록 ID와 메모를 유지한다.
- 같은 곡을 다시 포착하면 새 중복 기록을 만들지 않고 기존 개인 기록으로 안내한다.
- 사용자 화면에서는 카테고리를 나누지 않고 모든 태그를 하나의 개인 언어로 다룬다.
- 태그는 곡의 객관적인 장르보다 음악을 다시 찾고 싶은 상황·시기·관계·기억을 표현한다.
- 외부 장르, 감상 날짜, 등록일은 태그와 분리된 구조화 정보로 유지한다.
- 사용 중인 태그를 삭제하면 모든 곡의 기억에서 해당 태그가 함께 제거된다.

### 데이터 보호

- 메타데이터나 미리듣기 링크가 실패해도 사용자가 작성한 기억은 유지한다.
- 현재 데이터의 원본은 브라우저 `localStorage`다.
- 다른 탭에서 먼저 수정한 데이터가 있으면 이전 상태로 덮어쓰지 않고 최신 데이터를 다시 불러온다.
- JSON 백업·복원으로 브라우저 데이터 손실에 대비한다.

## 10. UX·디자인 방향

- 모바일 전용이며 앱 캔버스의 최대 폭은 430px다.
- 실제 모바일에서는 전체 화면을 사용하고, 넓은 브라우저에서는 같은 모바일 캔버스만 가운데 표시한다.
- 앨범 아트를 제품 사진처럼 크게 다루고 인터페이스 장식은 최소화한다.
- 흰색·옅은 회색·근접 검정 면과 단일 파란색 동작 색상으로 계층을 만든다.
- 챕터는 카드 대시보드가 아니라 앨범 커버와 편집된 목차로 표현한다.
- 모든 주요 터치 영역은 최소 44×44px를 확보한다.
- 화면 전환은 맥락을 유지하되 모션 감소 설정에서는 확대·스태거를 줄인다.
- 캐러셀, 다이얼로그, 플레이어는 키보드 포커스와 스크린리더 상태 안내를 제공한다.

자세한 시각 기준은 [design.md](../design.md), 모바일 검증 결과는 [design-qa.md](../design-qa.md)를 참고한다.

## 11. 현재 MVP 범위

### 구현 완료

- 로컬 개인 아카이브와 샘플·빈 상태 온보딩
- 곡만 기록과 태그 하나로 기록하기의 분리된 빠른 포착 흐름
- iTunes KR 검색과 Apple Music·Spotify·YouTube·Melon 링크 식별
- 자동 메타데이터가 부족할 때 제목·아티스트 수동 보완
- 수동·하위·자동 월별 챕터
- 태그 우선 곡 기억과 선택 상세 영역의 성격·시기·복수 날짜 메모
- 보관함, 통합 검색, 회고 3종, 태그 관리
- 30초 미리듣기와 원본 서비스 이동
- JSON 백업·복원과 오프라인 앱 셸
- 모션 감소, 포커스 관리, 모바일 터치 영역
- 시드 데이터 기반 공개 챕터 탐색, 공개·비공개 기록 표현, 프로필 팔로우·챕터 좋아요·활동 알림

### 부분 구현·검증용 기능

- 플레이리스트 내보내기는 3단계 UI와 모의 매칭만 제공한다. 실제 서비스 계정 연결·곡 매칭·플레이리스트 생성은 하지 않는다.
- 공개 챕터 탐색은 실제 계정·서버 없이 12개 이상의 가상 프로필과 로컬 상호작용 상태로 검증한다. 추천은 태그보다 곡 구성과 개인 아카이브의 공유 곡 수를 우선한다.
- 링크 메타데이터는 서비스가 공식적으로 제공하는 범위만 자동 완성한다. Spotify는 아티스트 보완이 필요할 수 있고 Melon은 수동 입력 방식이다.
- 오프라인에서는 선캐시된 HOME·CHAPTERS와 이전에 캐시된 화면에서 기존 기록을 볼 수 있지만, 새 음악 검색·링크 가져오기·미리듣기는 할 수 없다.

### 제공 범위 밖

- 계정, 로그인, 서버 데이터베이스, 기기 간 클라우드 동기화
- 실제 사용자 간 공개·비공개 공유, 댓글·DM, 서버 기반 팔로우·알림
- 직접 음원 업로드와 자체 스트리밍
- 실제 외부 서비스 플레이리스트 생성 API
- 네이티브 모바일 앱과 푸시 알림

## 12. 성공 기준

아래 값은 현재 제품에 계측된 성과가 아니라 다음 사용자 검증에서 확인할 목표다.

| 검증 항목 | 목표 |
|---|---|
| 첫 아카이브 완성 | 타깃 사용자 테스트의 80% 이상이 10분 안에 3곡에 개인 태그를 남기고, 그중 1곡을 수동 챕터로 옮긴다. |
| 입체적 기록 이해 | 사용자가 같은 곡을 두 챕터에 담고 서로 다른 태그·메모가 유지됨을 설명할 수 있다. |
| 재탐색 성공 | 기록한 곡을 챕터 탐색과 통합 검색 두 경로에서 모두 다시 찾는다. |
| 빠른 포착 회수 | 곡만 담은 기록과 태그만 남긴 기록을 구분하고, 나중에 챕터·상세 기억으로 확장할 수 있다. |
| 월별 분류 이해 | 자동 월별 챕터는 읽기 전용 등록 목차이고, 통합 검색·개인 기억 편집 대상이 아님을 이해한다. |
| 기록 가치 | 인터뷰에서 단순 좋아요·플레이리스트보다 기억 보존에 더 적합하다고 평가한다. |
| 디깅 전환 | 유사한 공개 챕터의 기록을 읽은 뒤 플레이리스트 내보내기와 프로필 팔로우 흐름을 완료한다. |
| 지속 사용 신호 | 2~4주 후 다시 방문해 과거 곡을 검색하거나 회고에서 여는 행동이 나타난다. |

## 13. 검증 계획

### 핵심 과업 테스트

1. 기억나는 곡 세 개로 `첫 자취방` 챕터를 만든다.
2. 같은 곡을 다른 챕터에도 기록해 서로 다른 성격·태그·메모를 남긴다.
3. 분류하기 어려운 곡은 보관함에 넣고 나중에 정리한다.
4. 곡명 없이 기억 문장이나 태그만으로 기록을 다시 찾는다.
5. 월별 챕터와 수동 챕터의 차이를 자신의 말로 설명한다.
6. JSON 백업을 만든 뒤 테스트 데이터로 복원한다.

### 인터뷰에서 확인할 질문

- 평소 좋아했던 곡을 놓치거나 잊은 경험이 얼마나 자주 발생하는가.
- 곡마다 성격과 태그를 붙이는 일이 가치 있는 수고로 느껴지는가.
- 챕터와 하위 챕터가 기존 플레이리스트보다 취향을 잘 표현하는가.
- 보관함이 기록 부담을 실제로 낮추는가.
- 어떤 순간에 검색이나 회고를 다시 열고 싶은가.
- 계정·동기화가 없다는 제약이 지속 사용을 막는가.

## 14. 우선순위와 후속 로드맵

### 1단계 — 핵심 수요 검증

- 실제 모바일 기기에서 첫 곡 기록부터 기억 기록까지 관찰한다.
- 태그·성격·시기 중 사용자가 실제로 반복 입력하는 필드를 확인한다.
- 보관함 회수율과 검색·회고 재방문 동기를 검증한다.
- 월별 챕터가 도움인지 중복인지 확인한다.

### 2단계 — 신뢰 가능한 개인 아카이브

- 계정과 서버 영속화 도입 여부를 검증 결과로 결정한다.
- 로컬 데이터의 클라우드 이전과 기기 간 동기화 정책을 설계한다.
- 초고속 포착을 위한 공유 시트·브라우저 확장 등 진입 경로를 검토한다.
- 실제 모바일 키보드, 안전 영역, 설치형 PWA를 반복 검증한다.

### 3단계 — 선택적 외부 연결

- 실제 플레이리스트 생성 수요가 확인된 서비스부터 OAuth·API 연동을 검토한다.
- 개인 기록을 훼손하지 않는 범위에서 챕터 공유나 내보내기 이미지를 검토한다.

## 15. 주요 위험과 대응

| 위험 | 현재 대응 | 추가 검증 |
|---|---|---|
| 기록 입력이 번거로움 | 보관함, 선택 입력, RECORD 검색·링크 초안 임시 저장 | 어떤 필드가 핵심 가치이고 어떤 필드가 이탈을 만드는지 관찰 |
| 로컬 데이터 손실 | JSON 백업·복원, 보호 모드 | 계정·동기화 도입 필요성 검증 |
| 외부 음악 정보 불안정 | 공식 메타데이터 범위 사용, 수동 보완, 원본 링크 유지 | 서비스별 성공률과 이탈률 측정 |
| 월별·수동 챕터 혼동 | 탭 분리, 연도 표기, 수동 배정 차단 | 사용자 언어로 역할 구분이 되는지 확인 |
| 계층이 과도하게 복잡해짐 | 루트 목록과 직접 자식 중심 표시 | 실제 평균 깊이와 탐색 실패 관찰 |
| 플레이리스트 기능 오인 | 연동 전 미리보기와 준비 완료로 명시 | 실제 연동의 지불·사용 의향 검증 |
| 제품이 일부 애호가에게만 유효 | 선명한 타깃을 우선 | 범용화보다 해당 집단의 반복 사용을 먼저 확인 |

# MUCHI 서비스 설명서

> 기준일: 2026-07-20
> 대상: 서비스 기획, 사용자 테스트, 개발 인수인계를 위해 MUCHI의 의도와 기능을 빠르게 이해해야 하는 사람

## 1. 서비스 정의

MUCHI, 뮤키는 좋아했던 음악을 잊기 전에 포착하고, 그 음악을 들었던 시기·장면·감정·문장을 나만의 언어로 쌓아 올리는 모바일 음악 아카이브다.

핵심 슬로건은 다음과 같다.

> 챕터로 쌓아 올린 나만의 음악 세계

뮤키는 일반적인 플레이리스트 앱이 아니다. 곡 목록을 만드는 것보다, 한 곡이 나에게 어떤 장면이었는지 기록하고 나중에 다시 꺼내는 데 집중한다.

## 2. 문제의식

스트리밍 서비스의 좋아요와 플레이리스트는 음악을 빠르게 저장하기에는 좋다. 하지만 시간이 지나면 다음 정보가 사라진다.

- 그 곡을 왜 좋아했는지
- 어느 시기와 장소에서 들었는지
- 어떤 감정이나 사람, 장면과 연결되어 있었는지
- 같은 곡이 다른 시기에는 어떻게 다르게 들렸는지

뮤키는 이 문제를 해결하기 위해 곡을 단순히 저장하지 않고, 사용자의 경험 태그와 메모를 함께 남긴다. 목표는 더 많은 곡을 모으게 하는 것이 아니라, 좋아했던 곡을 나중에 자신의 언어로 다시 찾게 하는 것이다.

## 3. 목표 사용자

뮤키의 1차 사용자는 음악을 많이 듣는 모든 사람이 아니다. 다음 행동이나 욕구가 있는 사람에게 더 선명하게 맞는다.

- 좋아요 목록 아래로 밀린 곡을 나중에 다시 찾고 싶은 사람
- 음악을 특정 시기, 계절, 사람, 장소와 함께 기억하는 사람
- 플레이리스트 제목 하나로는 곡의 의미를 충분히 담기 어렵다고 느끼는 사람
- 곡명이나 아티스트가 아니라 `스무 살 여름`, `새벽 러닝`, `첫 자취방` 같은 자기 언어로 음악을 찾고 싶은 사람
- 기록을 위해 최소한의 입력 수고를 감수할 의향이 있는 사람

우선 대상이 아닌 사용자는 단순 좋아요와 자동 추천만으로 충분한 사용자, 실시간 소셜이나 협업 플레이리스트가 주목적인 사용자, 음악을 개인적 맥락으로 다시 찾을 필요가 거의 없는 사용자다.

## 4. 핵심 원칙

### 빠르게 포착하고 나중에 확장한다

곡을 발견한 순간 모든 정보를 입력하게 하지 않는다. 사용자는 곡만 보관하거나, 태그 하나만 남기고 빠르게 기록을 시작할 수 있다. 챕터, 메모, 공개 여부, 추가 태그는 나중에 보완할 수 있다.

### 개인 태그 하나에서 기록이 시작된다

뮤키에서 태그는 장르나 정답형 감정 분류가 아니다. `운동할 때`, `문득 생각난 사람`, `겨울 지하철`처럼 미래의 내가 다시 찾을 수 있는 개인적인 단서다.

### 같은 곡도 맥락별로 독립된다

같은 곡을 여러 챕터에 담을 수 있다. 이때 곡 정보는 공유하지만 태그, 메모, 공개 상태는 각 챕터 안의 곡 기억마다 독립적으로 유지된다.

### 입력은 회수로 보상한다

기록의 보상은 저장 순간보다 나중에 다시 만나는 순간에 생긴다. 뮤키는 검색, 태그 그룹, 회고, 챕터 탐색을 통해 사용자가 남긴 단서를 다시 꺼내게 한다.

### 개인 아카이브가 중심이고, 탐색은 보조 축이다

타인의 공개 챕터를 둘러볼 수 있지만, 뮤키의 중심은 내 음악 세계를 만들고 다시 찾는 경험이다. 댓글, DM, 실시간 피드는 제공하지 않는다.

## 5. 핵심 개념

| 개념 | 의미 |
| --- | --- |
| 음악 | 제목, 아티스트, 앨범, 아트워크, 외부 서비스 링크를 가진 곡 정보 |
| 챕터 | 사용자가 직접 만든 음악 장면과 분류 단위 |
| 하위 챕터 | 챕터 안에서 더 작은 장면으로 나눈 구조 |
| 곡의 기억 | 특정 챕터 안에서 한 곡에 붙인 태그, 메모, 공개 상태 |
| 태그 | 미래에 다시 찾기 위한 개인 경험 단서 |
| 보관함 | 아직 챕터를 정하지 않은 곡의 임시 대기 공간 |
| 월별 챕터 | 뮤키에 처음 포착한 월 기준으로 자동 생성되는 읽기용 목차 |
| 뮤커 | 뮤키를 사용하는 사람 |

## 6. 주요 기능

### 6.1 로그인과 초기 설정

뮤키는 Google 로그인과 Supabase 저장소를 사용한다. 로그인 직후 사용자는 Google 프로필 이미지를 보며 닉네임을 설정한다. 닉네임에는 기본적인 글자수와 문자 규칙이 적용된다.

기록은 로그인한 계정에 저장되며, 브라우저 localStorage를 영구 데이터 원본으로 사용하지 않는다. 일부 입력 중 초안은 데이터 유실을 줄이기 위해 현재 세션에 임시 보존될 수 있다.

### 6.2 음악 추가

사용자는 하단 `기록` 탭에서 음악을 추가한다.

- iTunes KR 검색으로 곡명이나 아티스트를 검색한다.
- YouTube Music 또는 Apple Music 공유 링크로 곡을 가져온다.
- PWA로 설치한 경우, 지원되는 음악 앱의 공유 흐름에서 뮤키로 들어오는 사용 경험을 목표로 한다.
- 링크나 메타데이터가 부족하면 안내 메시지를 보여주고, 가능한 경우 직접 검색 흐름으로 돌아간다.

현재 공유 링크 자동 분석은 YouTube Music과 Apple Music을 지원한다. Spotify 가져오기는 준비 중이며, 베타에서는 직접 검색으로 같은 곡을 찾아 기록한다. iPhone/iPad에서는 음악 앱 공유 메뉴에서 뮤키를 직접 선택하는 대신, 곡 링크를 복사해 뮤키의 링크 가져오기 입력칸에 붙여 넣는다.

### 6.3 빠른 기록

곡을 찾은 뒤 사용자는 두 가지 방식으로 시작할 수 있다.

- 곡만 담기: 지금은 보관함에 넣고 나중에 정리한다.
- 태그로 기록: 태그 하나 이상을 남겨 바로 개인 기록으로 만든다.

태그로 기록한 곡은 챕터를 정하지 않아도 숨겨진 기록 컨텍스트에 저장된다. 이후 수동 챕터로 옮겨도 같은 기록의 태그와 메모가 유지된다.

### 6.4 곡 기억 편집

곡 기억 화면은 태그를 먼저 보여준다. 사용자는 이미 만든 태그를 바로 선택하거나, 새 태그를 입력해 추가할 수 있다. 태그가 많아지면 세 줄까지 먼저 보여주고, 필요할 때 펼쳐 전체 태그를 확인한다.

추가로 다음 정보를 남길 수 있다.

- 감상 날짜가 붙은 메모
- 여러 날짜의 메모 누적
- 태그와 메모의 공개/비공개 상태
- 다른 챕터로 이동하거나 챕터 안에서 삭제

태그와 메모 공개 여부는 하나의 단위로 묶는다. 공개하면 탐색이나 공개 챕터에서 함께 보이고, 비공개하면 곡은 보이더라도 기록 내용은 숨겨진다.

### 6.5 챕터와 하위 챕터

챕터는 사용자가 직접 만든 음악 세계의 목차다.

- 새 챕터 생성
- 대표 이미지 설정
- 공개/비공개 설정
- 하위 챕터 생성
- 챕터 안 곡 목록 관리
- 챕터별 플레이리스트 내보내기 흐름 진입

대표 이미지를 설정하면 해당 이미지가 챕터 커버 전체를 채운다. 대표 이미지가 없을 때는 수록곡의 앨범 아트가 분할되어 표시된다.

챕터 공개 여부는 탐색 피드와 방문자 화면에 노출될지 결정한다. 챕터가 공개여도 각 곡의 태그와 메모가 비공개이면 기록 내용은 노출되지 않는다.

### 6.6 보관함

보관함은 좋아요처럼 빠르게 곡을 놓치지 않기 위한 대기 공간이다.

보관함에서는 각 곡을 바로 삭제하거나 기록 흐름으로 보낼 수 있다. 사용자는 아직 정리하지 않은 곡을 챕터로 옮기고, 태그와 메모를 남겨 기록을 완성한다.

### 6.7 태그

태그는 뮤키의 핵심 회수 수단이다.

- 모든 태그는 하나의 개인 언어 라이브러리로 관리한다.
- 태그 카테고리는 사용자 화면에 노출하지 않는다.
- 태그 이름은 최대 30자다.
- 태그 선택 UI는 곡 기록과 곡 찾기에서 동일한 인라인 구조를 사용한다.
- 새 태그는 별도 모달 없이 태그 목록 앞에서 바로 만들 수 있다.
- 태그 관리 페이지에서는 태그 검색, 수정, 삭제를 할 수 있다.

태그는 관리 기능 자체보다 음악을 다시 찾기 위한 인터페이스다. 사용자가 태그를 누르면 해당 태그가 붙은 곡과 챕터별 기억을 다시 볼 수 있다.

### 6.8 곡 찾기

`곡 찾기` 탭은 내 아카이브를 회수하는 화면이다.

- 곡명, 아티스트, 챕터, 태그, 메모를 통합 검색한다.
- 여러 태그를 조합해 `모두` 또는 `하나 이상` 조건으로 찾는다.
- 검색 결과의 태그를 누르면 같은 태그 조건으로 다시 검색한다.
- 같은 곡이 여러 챕터에 있으면 각 챕터의 기억이 별도 결과로 나타난다.

검색 화면의 태그 UI는 곡 기록 화면과 같은 방식으로 동작한다. 태그를 바로 고르고, 새 태그를 만들고, 세 줄 초과 시 펼쳐 볼 수 있다.

### 6.9 회고

회고는 기록의 보상을 다시 돌려주는 기능이다.

- 이맘때의 음악
- 타임라인
- 무작위 기억

회고는 통계 대시보드가 아니라, 과거에 남긴 음악과 메모를 다시 꺼내는 경험이다. 자동 월별 챕터는 등록 시점의 목차이고, 감정적 회고는 사용자가 남긴 태그와 메모의 맥락을 우선한다.

### 6.10 탐색

탐색은 다른 뮤커가 공개한 챕터를 발견하는 공간이다.

- 공개 챕터 피드
- 공개 챕터 상세
- 공개된 태그와 메모 열람
- 비공개 기록 안내
- 프로필 팔로우
- 챕터 좋아요
- 팔로잉한 뮤커의 새 공개 챕터 활동

추천의 중심은 태그 일치가 아니라 곡 구성과 내 아카이브에 이미 있는 곡의 겹침이다. 태그는 타인의 챕터를 이해하는 보조 맥락으로 사용한다.

탐색은 소셜 네트워크가 아니라 공개 아카이브 디깅에 가깝다. 댓글, DM, 리포스트, 실시간 대화는 제공하지 않는다.

### 6.11 플레이리스트 내보내기 미리보기

챕터에 담긴 곡은 Apple Music, Spotify, YouTube Music을 대상으로 한 내보내기 흐름을 열 수 있다.

현재는 실제 스트리밍 계정에 플레이리스트를 생성하지 않는다. 사용자는 다음 흐름을 미리 경험한다.

1. 내보낼 챕터와 곡 확인
2. 대상 서비스 선택
3. 모의 매칭 결과 확인
4. 내보내기 준비 완료 후 외부 서비스 열기

이 기능은 실제 API 연동 전, 사용자가 챕터를 듣기 흐름으로 전환하고 싶어 하는지 검증하기 위한 것이다.

### 6.12 백업과 데이터 보호

뮤키의 원본 데이터는 Supabase에 저장된다. 저장 실패가 발생하면 변경사항을 즉시 버리지 않고, 가능한 범위에서 보존한 뒤 재시도 안내를 한다.

설정에서는 JSON 백업을 제공한다. 백업은 계정 저장과 별개로 사용자가 자신의 기록을 직접 보관하기 위한 안전장치다.

## 7. 공개 범위 모델

뮤키의 공개 설정은 두 단계다.

| 단위 | 설정 | 공개 화면 동작 |
| --- | --- | --- |
| 챕터 | 공개 / 비공개 | 공개 챕터만 탐색과 공개 프로필에 노출 |
| 곡의 기억 | 태그/메모 공개 / 비공개 | 공개 시 태그와 메모가 함께 노출, 비공개 시 곡만 보이고 기록 내용은 숨김 |

메모만 공개하거나 태그만 공개하는 방식은 제공하지 않는다. 태그와 메모는 사용자가 남긴 하나의 기록 단위로 본다.

## 8. 현재 제공하지 않는 것

뮤키는 현재 다음 기능을 제공하지 않는다.

- 자체 음원 스트리밍
- 직접 음원 업로드
- 실제 Apple Music, Spotify, YouTube Music 플레이리스트 생성 API 연동
- 댓글, DM, 방명록, 리포스트
- 네이티브 푸시 알림
- 협업 플레이리스트
- AI 자동 태그 생성
- 태그 트렌드나 인기 태그 기반 추천
- 데스크톱 전용 레이아웃

이 기능들은 불가능해서 제외한 것이 아니라, 현재 검증해야 하는 핵심이 `개인 음악 기록과 재발견`이기 때문에 후순위로 둔다.

## 9. 제품 판단 기준

뮤키가 성공했는지는 기능 수가 아니라 다음 행동으로 판단한다.

- 사용자가 음악 앱에서 발견한 곡을 뮤키에 빠르게 저장하는가
- 첫 곡에 자기 언어의 태그를 남기는가
- 태그와 메모를 남기는 일이 부담보다 가치 있게 느껴지는가
- 며칠 뒤 곡명 없이 태그나 기억 문장으로 다시 찾는가
- 같은 곡을 여러 챕터에 담아도 각 맥락이 독립된다는 점을 이해하는가
- 보관함에 둔 곡을 다시 정리하는가
- 회고나 탐색을 통해 기록을 다시 열어보는가

현재 제품의 다음 검증 중심은 새로운 기능 확장이 아니라, 실제 사용자가 1~2주 동안 곡을 기록하고 다시 찾는지 확인하는 것이다.

## 10. 기술·운영 범위

- 프레임워크: Next.js, React, TypeScript
- 인증: Supabase Auth, Google 로그인
- 데이터 저장: Supabase PostgreSQL
- 외부 음악 정보: iTunes 검색, YouTube Music·Apple Music 공유 링크 메타데이터
- 앱 형태: 모바일 기준 PWA 웹앱
- 배포 대상: Vercel
- 오프라인: 제한적 앱 셸 캐시

앱은 모바일 경험을 기준으로 설계한다. 실제 모바일에서는 전체 화면을 사용하고, 넓은 브라우저에서는 최대 430px 모바일 캔버스 안에서 동작한다.
